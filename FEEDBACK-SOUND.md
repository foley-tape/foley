## 结论一句话

SOUND-R1 全部落地：声音自 probe 生成页析出为 `sound/` 模块（音频图注册表＋方案 B 呼吸乘法级），机器耳朵 `cli ear` 上岗——**G1–G3 active 全绿（G5 超线 3.2dB，G4 如实记分附归因）**，probe 薄壳化（页内手抄图代码物种灭绝），DONE 断电落针落地且五带判定无触动。金测试 **62/62 绿**（新增声音相 ㊱–㊸ 与引擎 ㊹）。sound-params `9e2b3a06 → eb050e56`（新增 `bed.breathDepth` 0.12）。**按 §4 发起一次实听申请（见文末）。**

## 一、修法落点（§2 方案 B ＋ 注册表三规矩）

- **方案 B 呼吸乘法级**：`正身 → s1Breath(bias=1, ±breathDepth) → s1Level → bedBus`。双肺 LFO（1/7.3、1/11.9Hz，Eno 互质保留）以 0.6/0.4 分深摆动乘法级，**数学上永远是正身的百分比**——反相、压过正身两种事故形态结构性灭绝。金测试 ㊳：渲染包络实测被夹在 正身×(1±0.12) 内；depth=0 时床静止。
- **注册表**（`sound/graph.js`，拓扑图纸在文件头注释）：
  ① 一切节点经注册表出生；② `stopAll` 遍历电平＋调制深度参数（撤单→0.05tc 归零→**+300ms 自动化硬闸**，EAR-3 的 JS 定时器硬闸自动化化，离线可渲染），一次性源（前景/S2 打点）全部登记、停止当场枪毙——僵尸结构性灭绝；③ 外接调制唯一通路 = `modulate()` 乘法级/调制口（s1Breath.gain、wowDelay.delayTime，均 bias±有界深度），**`connect` 见 AudioParam 即抛**（金测试 ㊲）。
  trim 不是独立通路：它在 `core.bedTargets` 内乘进每一个电平目标、applyBed 是电平唯一写者——"绕过总闸"的支路无处出生（EAR-2 房间噪绕闸一类的结构性灭绝）。
- **同源的结构化**：`sound/core.js`（纯映射律）与 `sound/graph.js`（音频图）为纯 JS 单一事实源——probe 页**逐字内嵌真源**（构建时剥模块语法），`cli ear` 离线渲染 import 同一份文件。"页里手抄同源律"（EAR-4 失明土壤）从格局上不再可能。TS 类型面走 `core.d.ts / graph.d.ts`，老进口（`sound/index.ts`）路径命名不变。

- 渲染器 `sound/offline.ts`：Web Audio 语义子集纯 TS 实现（零依赖）。**保真命门 = AudioParam 计算值 = 内在自动化值＋外接信号之和，而 `.value` 只报内在值**——金测试 ㊱ 以 EAR-4 事故拓扑原样重演，渲染比账本响 +10dB、账本仍读 0.035：耳朵自证不受账本蒙蔽。`.value` 读数自此在验收链路中零地位（probe 页 `__probe.gains()` 已标注"仅 dev 展示"）。
- G1 含阳性对照（㊵：不停止同窗 > −45dBFS 有声）与预排前景猎杀（停止前已排到未来的拨弦/ASK 同殁）。
- G3 口径：bedBus 点（S4 磁带总线染色不入账），设计值 = `core.bedRmsDb`（见现实修正④ 定标轮）。
- 产物：`runs/ear-machine-<ts>/EAR_MACHINE.md`，携四哈希；`runs prune` 规约内。
