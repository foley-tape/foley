# TAPE-0 施工方手记 · M1.5

> 这是我（施工方）的**主观**思考，与 `FEEDBACK.md`（客观交接）分开。
> 目的：把执行中撞见的、**超出施工令范围**的判断和张力摆上台面，替你提前排雷。

## 一、复盘：这轮我做错 / 学到的三件事

1. **我按"简历"荐卷，没按"表现"荐卷。**
   上轮我把 `4a0286c7` 荐为 storm，凭的是体检表（308 事件、活跃 224min、"连贯"），却没真喂进引擎看张力。结果它峰值才 0.3、是卷顺风会话。
   **教训（已固化为流程）：任何标准带在提名前必须先 replay 过引擎——体检表是必要不充分，引擎判定才是真相。** 这也是我这轮先把 6 个候选全部回放、再定卷的原因。

2. **先造"法官"，再信"调参"。**
   真正救场的不是引擎，是 §6 的验收自动判定——它把"感觉对不对"变成机器判定，一次回放就逮住了我的误选。**判据（法官）比参数（旋钮）更该先立稳。**

3. **参数是耦合的，而我们没有耦合图。**
   你这轮把 READ 权重 0.2→0.12（一个合理的"读别太吵"的通用决定），却悄悄掐死了以读失败为主的风暴带。**没有"每条带依赖哪些参数"的映射，任何一次调参都可能远程打断另一条带。** 建议维护一张极简耦合表（我可以生成）。

## 二、我看到、但没擅动的优化（不在施工令内，留给你裁）

1. **RESOLVE 太窄——这正是 storm RESOLVE=0 的深层原因。**
   现在"如释重负"只由 test-OK 触发。但真实的解脱还来自：卡碟解除（STUCK_CLEARED）、憋很久的构建终于过、苦战后的一次提交。**建议把"泄能＋下坠＋声音"从"仅测试通过"拓宽到这几类**，否则 READ 风暴永远等不到那口气。

2. **占空比可能是把错的风暴尺子。**
   风暴是**阵发**的（一簇簇尖峰），不是**持续**下雨。"RAIN+STORM ≥15% 时间"对真实编码风暴可能结构性不可达。**换尺子？** 比如"峰值触到 STORM ≥N 次"或"T 超阈的积分面积"，比时间占空更贴风暴本性。

3. **也许风暴不止一种质地。**
   "破坏型"（RUN/WRITE 失败＝真出血）vs"迷路型"（READ 失败＝疯狂翻找）。`1bb942f0` 是**迷路型**。你降 READ 权重对"破坏型"也许对，只是误伤了"迷路型"。**要不要分别建模？**（也牵动三音——迷路和出血该不该是同一种声音？）

4. **errClass 是蒸馏带最后一处隐私残留。**
   蒸馏带已"接近可分享形态"，唯一文本字段就是归一化的错误首行。**要更彻底可把它也哈希掉**（只留聚类 id），换完全可分享——代价是报告里读不到人类可读的错误模板。留可读、还是求可分享？

5. **episode 硬复位把多日风暴切成了一坐一坐的弧。**
   现在跨段 S 归零。但人 resume 一场苦战时，情绪是有残留的。v0 硬复位够用，但这是个有叙事代价的建模选择——**要不要软复位（跨段留一点余温）？**

6. **replay 与 live 的 reap 时机会有微小分歧（给 M2 提前示警）。**
   卡碟窗口过期的 STUCK_CLEARED，回放里在"驱动越过过期点"时精确发；直播里取决于 tick 频率。M2 接线时要对齐，否则同一带 replay 与 live 的 CLEARED 时刻会差几百毫秒。

## 三、给你的更大决策点（现在定，省得以后返工，按"越晚越贵"排序）

| # | 问题 | 不定的后果 |
|---|---|---|
| 1 | **风暴是一种还是几种**（破坏/迷路/卡碟）？ | 决定权重、尺子、三音；定得晚要重调所有带 |
| 2 | **解脱（RESOLVE 类）要不要拓宽**到 CLEARED／憋久的成功／苦战后提交？ | 决定引擎泄能逻辑＋M2"释放音"，越到 M2 越难加 |
| 3 | **验收用"区间带"还是"金曲线"**？你审批一条"风暴该长这样"的参考曲线、我拟合参数；还是继续猜区间？ | 决定整个校准流程的形态 |
| 4 | **验收判据归谁**？现在硬编码在 `replay.ts`。当参数（可版本化/可调）还是当规范（冻结）？ | 判据漂移无人签核的风险 |
| 5 | **M2 的 STUCK_CLEARED 要不要有声音**？§8 三音没给它。 | 与 #2 联动，M2 起手就得定 |

**我的倾向**：#1 先定（它是根），#2 紧随；#3 若你愿意给一条金曲线，后续每轮校准会快十倍。
