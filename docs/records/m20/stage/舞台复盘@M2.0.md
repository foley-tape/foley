## 0. 一条脊柱

三轮走完回头看，舞台侧真正的设计原则只有一条，且是逐轮才悟全的：

**引擎管物理，舞台管光学。凡在舞台上私造物理，都是对磁带说谎。**

- M-S1 交出了缓动：针的全部生命力来自引擎的弹簧-阻尼数据，渲染端只做两包线性重建。
- M-S2 交出了节拍：广播环从 rAF 改间隔钟——显示器的钟不配给磁带定拍。
- M-S3 交出了钟本身：live 客户端没有本地时间轴，stageT = t − t₀，钟在包里。

一轮比一轮少拿一样东西。今天舞台手里只剩三样：光、材质、恒迟 50ms 的重建。我认为这条脊柱值得写进某份正典（宪法或白皮书的画侧总纲），因为预告片轮会有新人手（或新代理）进场，他们的第一冲动一定是给针加 transition——那是前端的肌肉记忆，得有条文压着。

## 1. 没人点名、但我认为分量最重的几笔（已做）

这些都不在令上，也不在宪法里，是施工中自己长出来的。回头看，它们承担的重量超过了当时的体积：

**① 墨点记纸位，不记时刻。**
走纸的数据结构选了"纸空间"而非"时间空间"——每个墨点存它落在纸上的位置。由此免费得到三件事：8× 铺纸与 1× 实走在纸上**全等**（屏录预热不留指纹）；DONE 纸停天然正确（纸不走，点不挪）；接带痕有处可钉。**教训的一般形**：叙事性器件（纸带即时间轴）的存储坐标应该选"介质坐标"而非"时钟坐标"，一次选对，诚实是白送的。

**② 恒迟一包的重建。**
"禁自加缓动"与"60fps 渲染 20Hz 数据"之间有一道窄门：两包线性重建，显示恒迟一包（50ms）。无过冲、无滞后动力学、迟得有界——恰好落在时间法"宁迟勿早"的下限上。这个模式三轮没改过一行，回放与 live 通吃。

**③ 双钟分频，先于宪法。**
灯与呼吸走真实时间、走带机构走舞台时间——M-S1 凭直觉做了，M-S2 手记里写"这条没写进宪法，但它应该是对的"，v1.2 入宪。复盘想说的是方法论：**当一个实现决定让你觉得"机器在快进但它对人的语速不变"这种话说得出口，它就该提请入宪**——品味执照的正确用法是先斩后奏之外的第三条路：斩了、奏了、等追认。

**④ 计数轮由 ∫ω dt 驱动。**
数字轮不吃时钟，吃带轴角速度的积分。带停轮停、倍速轮快、DONE 后一格不多走；千位轮"低位跑完最后一成才动身"的进位质感是公式白送的。**一般形**：器件间耦合宁走物理量（ω、纸位），不走全局钟——每少一处读钟，倍速与暂停就少一处特判。

**⑤ 自举验证。**
M-S3 让 live 尾随建造它的这场会话。这不是炫技：它是素材诚实条款在工程层的最强执行——**不造假带，连验证带都不造**。引擎判我 T=0.067 那一刻，等于项目第一次用自己的仪器给自己量了体温。建议保留为传统：每轮接线验证，首选拿当轮会话自己当带。

**⑥ 把"应该消失"翻译成数字。**
"墨线阶梯应自然消失"是句品味话，我把它翻成了平台占比（相邻墨点 ΔT=0 的比率）——然后它立刻回咬了我一口（见 §2）。翻译本身是对的：**每句品味验收都值得配一个机器可判的影子指标**，白皮书 §6 的声侧已这么做（Pearson r ≥ 0.6），画侧建议补齐同类条目（如：针尖峰速上限、墨线二阶差分谱、恒迟实测 ≤ 60ms）。

## 2. 撞过的墙（与每面墙上的字）

**墙一：藏起的标签页里，rAF 整个冻住。** 回放钟睡死，playing=true 而 stageT 不走。修法是广播环改间隔钟。墙上的字：**凡叫"钟"的东西，问一遍它在谁的手里**——rAF 在合成器手里，setInterval 在节流器手里，只有包里的 t 在引擎手里。M-S3 的"钟在包里"是这面墙教的。

**墙二：录像上下文里，画布先量了尺寸、样式后到。** Playwright recordVideo 的加载时序让 ChartRecorder 在样式表上身前量了 getBoundingClientRect，画布只画了左上四分之一。修法：boot 等 `load`。墙上的字：**先穿衣再量体**；一切"构造时量布局"的组件都欠一个 readyState 检查。

**墙三：SSE 让 networkidle 永不成立。** live 页挂着长连接，Playwright 等 networkidle 等到超时。墙上的字：验证工具的等待条件也是被测系统的一部分。已记入手记省后人半小时。

**墙四：平台占比在 replay 侧也是 0。** 我预言 10Hz 重建 ≈50% 平台、20Hz ≈0%，结果两边都是 0——因为 M-S1 的线性重建早把阶梯抹了，我的对照组被自己的历史修复污染了。这是三轮里最有营养的一跤：**验证设计要先问"对照组还干净吗"**。最终报告改口为诚实版本：真 20Hz 的收益不是"变平滑"，是"平滑从插值的猜测变成数据的事实"。

**墙五：jam 带里没有卡碟。** 施工令给 jam 的台词（"同槽反复"）在封版数据里没有实景，STUCK 四起全在 storm。没替它演——这一跤直接催生了素材诚实条款（M2.0 §0.2）。墙上的字：**令文描述的是期望，带子记录的是事实；两者冲突时，带子赢**。

**墙六（未撞上，但听见了风声）：端口与多轮会话。** 4173 被上一轮的服务占住，靠 PORT 环境变量 + autoPort 绕开。小事，但预示了 live 常驻化之后的续命问题（见 §4 探讨五）。

## 3. 没人点名、我认为最值得做的（未做，按值钱程度排序）

**① 今晨的纸（replay 前缀 + live 尾巴）。**
现状：live 开机即"现在"，纸从空白开始——**一台以"纸带即时间轴"立论的机器，每次开机都失忆，这是舞台当前最大的论题自伤**。而材料其实齐了：`cli live --out` 已在持续写 curve.csv/moments.csv（bounded，追加即丢）。舞台侧可以纯在 stage/ 围栏内实现：开机先以回放管线吃当日 out 文件铺纸（8× 铺纸与实走全等，§1① 的红利），追平后无缝切 SSE。零引擎改动，零协议改动。这与预告片轮的"高光选段"是近亲但不同物：这是**产品的记忆**，那是**素材的剪辑**。建议列入 v1.1 首件。

**② 机器的体温（能耗纪律）。**
这台机器的愿景是"陪一下午"。现状：镜头 shader 全屏 60fps 重绘、走纸每帧全量重画、漂移持续逼合成器工作。**没量过，该量**——但可以预判三刀：镜头降 30fps（胶片本来就是 24fps 的艺术，颗粒 30fps 只会更像电影）；走纸改增量绘制（墨是只追加的：上一帧平移一段 + 补画新墨条 + 笔桥，脏矩形天然成立）；`document.hidden` 时渲染整体停摆（状态照走，M-S3 已保证——看不见时不画，宪法级正确）。**风扇转起来的那一刻，声音相的整个床就毁了**——功耗不是工程卫生，是音画设计的一部分。建议 M-S4 给一条"体温预算"（如：常驻 CPU < 5%，能耗面板绿档）并入验收。

**③ weather 的画侧最小表达。**
StatePacket 九个字段，画侧今天只剩 weather 一个全哑。绑定表写着"天色/氛围档（若做窗）"——不必做窗：**画外那盏灯本来就有档位**。雨档灯温降一档、亮度收一成，风暴档再降，全部小节边界切换（与声侧同拍）。若不做，盲听 v2 的耳朵会听见雨、眼睛却看不见任何变化——音画同源同钟，一侧哑着，另一侧的诚实也会被质疑。实现约一天，光法条文内自由（只压暗，不提亮）。附带一个宪法问题，见 §4 探讨三。

**④ 性格照管线转正为视觉回归。**
三轮下来，捕捉脚本事实上已是回归工具：同带、同窗口、同秒——任何宪法/材质改动后重拍，逐帧可比。建议转正：`stage/portraits.json` 存五带窗口定义（带名、prefill、起点、秒数），一条命令全量重拍；架构师验帧就有了"上一轮 vs 这一轮"的固定机位。素材诚实条款顺便机械化——窗口定义在册，谁也没法临时挑好看的一段。

**⑤ IDLE 的深睡台阶。**
现状 IDLE 只压灯一档 + 带轴停。"陪一下午"的机器该有更深的睡姿：IDLE 持续 5 分钟后再降一档、呼吸性暗角放慢、只剩待机粒可辨——被叫醒时（新事件）灯用钨丝速度爬回来。全在光法与动法条文内，半天工作量，把"洗碗机式寂静"补成"深夜车间"。

## 4. 值得往下谈的（议题，非提案）

**探讨一：DONE 之后，针的余读。**
封版数据里 storm 的 DONE 段 needle 仍有 0.03 并以分钟级缓慢归零——编舞要"针归零"，数据给的是"针在磨蹭"。舞台按诚实条款如实渲染，不清零。但这值得引擎侧想一想：DONE 时刻发条是否该断电（needle 目标位直接置 0，靠既有弹簧-阻尼自然落针）？物理上说得通（马达断电、表头失励磁），戏剧上"洗碗机时刻"会更干净。**跨栏观察，不伸手**——留给 Track-FIX 与你裁。

**探讨二：fixtures 的 pendingAsk 清账时机。**
M1.9 §1.2 的增列 Track-FIX 已交（写出器已带 pendingAsk 列），但五带 runs/ 产物是封版当刻生成的八列旧档，舞台回放模式因此还在用 WAITING⇔pendingAsk 推导（M-S1 现实修正二）。缺的只是一次五带重跑。两个走法请裁：等 Track-FIX 下次例行重跑后知会我刷新副本；或**准我用封版 CLI 只读重放五带、产物直接落舞台自己的 fixtures/**（tapes/ 只读、runs/ 不碰、封版哈希保证逐字节同源）——十分钟的活，删掉推导，少一处未来的分叉口。

**探讨三：weather 上灯之后，灯归谁管。**
若探讨/提案③落地，画外灯就成了"承载变量的器件"——器件法（每件最多两义）管不管它？我的倾向：**光法与灯位法管灯，器件法管仪表**；画外灯是布景不是器件，weather 是氛围不是读数。但这条界线值得你亲自画——预告片轮会有更多"氛围承载信息"的诱惑（配乐、色温、景深），界线不画，器件法迟早被绕空。

**探讨四：发布素材的颗粒——一次成像还是后期。**
镜头 shader 把颗粒/暗角烤进每一帧，屏录即成片（一次成像，磁带宇宙的做派）。但动态颗粒对视频压缩是灾难（crf 18 下颗粒抖动吃码率；hero GIF 会更惨）。预告片轮需要决断：发布物料走"RAW 拍摄（镜头层关闭）+ 后期统一调色压片"，还是坚持一次成像、接受体积？我的品味票投一次成像，但工程账（GIF 尺寸、CDN）你算了才算。折中案：GIF 类导出用静态颗粒单帧化。

**探讨五：live 的常驻化与多会话。**
今天 live 是"一次 spawn 尾随一卷"；真实的一天是多终端并行（本项目自己就是双轨）。cli live 单文件尾随的架构下，舞台若要看"整个车间"，是多 spawn 多机架（一会话一台机器，房间隐喻天然可扩），还是引擎侧做多带混流？前者全在舞台围栏内但吃体温预算，后者动协议。不急裁——但预告片轮若想拍"双轨同开"的镜头，这题就到期了。

**探讨六：交互宪法，赶在撕纸带之前。**
预告片轮预告了"撕纸带交互"。舞台迄今唯一的交互是 loupe 悬停——一个"凑近看"的镜头动作。建议在新交互进场前立一条：**一切交互必须是这台七十年代机器的主人会做的动作**（凑近、撕纸、轻敲表头玻璃），任何"界面味"的交互（按钮、菜单、拖拽滑块）不得上面板——dev 抽屉除外。一句话的修正案，挡住十个未来的坏主意。

## 5. 收束

三轮的手感浓缩成一句话：**这台机器每交出一样东西（缓动、节拍、时钟），就更诚实一分；它每多记住一样东西（纸位、角度、包），就更像一台机器一分。** 下一轮无论先做"今晨的纸"还是预告片，我都建议先把 §0 那条脊柱写进正典——舞台以后不会永远只有我一个施工方。
