# 致架构师 · M2.1 后六个提点（Track-STAGE）

> 性质同《舞台复盘》续篇：施工中长出来的东西，非本轮交付清单项。一页纸，每点两三句。

**一、治理缺口：交付与合并之间有一段幽灵期。**
本轮实证：M2.1 全部交付提交在 stage/amber 上，桌面已按令清空，而 main 未收——船长眼里东西"消失"了整整一段对话。三轨并行后 main 移动频繁（本轮我 rebase 四次），"何时合并"却无成文。**提案一句入治理**：轨分支交付即自合并（FIX 轨早有先例），FEEDBACK 落笔必须记录已合入的 main 哈希——"交付"的定义从"提交在轨上"改为"船长在 main 上看得见"。

**二、影子指标的自反性：首轮影子先咬自己，是特性不是事故。**
三条影子，两条先揪出测量工具的病（倍速段两钟混采、追包补发时光学被迫超速），一条真修了产品（体温）。**提案**：§2 推广时把"影子的影子"写进流程——每条影子先在已知条件下验证读数符合构造预期，再上岗采数。验收工具与被验对象同等对待，这是审计夜"预注册"精神在画侧的延续。

**三、恒迟不是统计量，是恒等式——可直升 active。**
渲染端 delay = (now−realT₁) + 50·(1−frac)，在 frac<1 时**数学上恒等于 50ms**；实测 p50=p95=50.0 只是恒等式的回声。总纲"渲染只做恒迟一包的重建"因此有了机器可验的形态。**提案**：此条影子跳过"两轮观察期"直转 active（对恒等式做统计观察是对数学的不敬）；针速比与体温两条照旧候期。

**四、体温该记"像素账"，不是笼统 CPU%。**
7.71%→3.62% 的解剖：大头不在逻辑在像素——全屏原生分辨率 shader 每帧触碰 4M 设备像素，占余温近半；走纸增量化后每帧只碰一条 26px 的新纸。**提案**：体温预算按层拆"每帧触像素数"记账（镜头/走纸/器件/合成各一行），比 CPU% 更可设计、可预算、可跨机对比——并直接服务提点五。

**五、多机架终裁的实数（§0.6 要的体温数据，先给单机侧）。**
单机绿档 3.62%，其中镜头层约半。若多机架=N×整套，四机架≈9%+，必超预算；但镜头（颗粒/暗角/漂移）与房间光本就该**一室一份**——N 台机器同在一个房间、一颗镜头下。**结论供终裁**：多 spawn 多机架在"共享镜头层+共享房间光"的架构下才立得住，预算 ≈ 3.6% + (N−1)×约1.8%；若各机各带镜头，此路不通。

**六、昨日的卷：今晨的纸的免费续集。**
`cli live --out` 的产物若按日轮转（runs/live-<date>/），昨日目录天然就是一盘完整"日带"——舞台加一个 `?tape=yesterday` 回放入口，零新机制（读 out 产物的管线今天已在跑）。这同时是预告片轮"高光选段"的原料仓：选段算法吃的就是日带。**提请预告片设计案顺手定下日带轮转命名**，舞台侧半天接通。
