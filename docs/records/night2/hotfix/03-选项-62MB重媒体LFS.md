# 03 · 选项书：62MB 重媒体物理墙

> 候架构师裁「重媒体归档策」（M2.3 记案已加急）。

## 现场

M-T2 三支真接带 mp4 已提交进 `docs/records/mt2/stage/` 并推送成功：

| 文件 | 体积 |
|---|---|
| `foley-dub-2026-07-05-2026-07-06.mp4` | 63MB |
| `foley-dub-jam-2026-07-06.mp4` | 62MB |
| `foley-dub-storm-2026-07-06.mp4` | 62MB |

合计 ~190MB，另有三支 poster.png（各 ~2MB）。

- GitHub 反应是 **>50MB 建议警告**，非 >100MB 硬墙——目前「贴了罚单没扣车」，已推成功。
- 但已**永久进 git 历史**：每次克隆一次性 +190MB，且策不立、下轮物料只增。
- 三支 mp4 是 1080p30、9.1× 实时导出的「正片」存档。

## 选项

| 选项 | 做法 | 代价 | 风险 |
|---|---|---|---|
| L-1 既往不咎＋前向立规 | 三支留历史（未过硬墙）；立规「>50MB 禁直提」＋ pre-commit/CI 尺寸闸 | ~零 | 克隆体积不回收；只防将来 |
| **L-2 出库归档（发布物料正解·推荐）** | 大媒体走 **GitHub Releases 附件**（单件 2GB，免费不限额）或本地归档盘；仓库只留 `sha256+来源` 指针清单 | 中：定指针清单格式 | 无；Releases 本就是发布物料的家 |
| L-3 Git LFS 迁移 | `git lfs migrate import --include="*.mp4"` 重写历史 | **高**：强推＋所有工作台重克隆；LFS 免费仅 1GB 存储/月 1GB 带宽，超即收费 | **高**：本项目多并发会话＋双工作台（tape0／tape0-stage），历史重写是踩踏高发区；190MB 已吃免费额度近两成 |
| L-4 治本于源 | 出片降档：720p 或两遍 VP9，62MB→约 15–20MB | 低 | 「正片」存档质量是产品口味，须架构师/船长拍板 |

## 建议

近期 **L-1 ＋ L-2**：三支不动，立尺寸闸＋新物料一律走 Releases 指针。**L-3 不建议**——除非架构师决意长期在仓库内收媒体，否则在多并发会话＋双工作台环境下，历史重写不值这个险。**L-4** 可作 L-2 的叠加项一并递交（决定「正片」是否降档存档）。

## 关联

- `private:true` 拆闸时须定 `files` 白名单，把 `docs/records/` 重媒体挡在 npm 发布包外——与本案同源，见 01 件末。
