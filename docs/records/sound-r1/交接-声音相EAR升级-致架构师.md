> 性质：**故障升级 + 施工方自审**。船长四轮实听均报"床是大噪声"，我三轮修复三轮"验证通过"，全部与船长耳朵矛盾。第四轮船长叫停迭代、令升级。已审计出根因（有实证），**按令未再动代码**，修法与验收协议升级案在此候裁。

## 0. TL;DR

床里最响的声源是一处**从第一版就存在的错误连线**：S1 呼吸 LFO 以绝对深度 ±0.25 直连 S1 增益参数（设计电平仅 0.035）——**副肺比正身响 17dB**。它同时解释：床=大噪声、trimDb 拖动无效、停止后仍响、音乐退场噪声留下。我三轮修复修的都是真 bug，但都不是这只；三轮"验证通过"全因我的表**结构性失明**（`.value` 读数不含外接信号贡献）。修法一行可改，**候你裁**；更要紧的是验收协议要升级成"机器实听"，否则此类 bug 永远能穿过我的验证。

## 2. 根因（代码级，可复核）

`cli/probe.ts` 生成页 S1 段（自声音相第一版起未变）：

```js
bg1.gain.value=0.15; bg2.gain.value=0.1;                    // ← 绝对深度
b1.connect(bg1); bg1.connect(G.s1.gain);                    // ← LFO 直连增益参数
b2.connect(bg2); bg2.connect(G.s1.gain);
```

- 设计意图：Eno 式互质周期（7.3s/11.9s）呼吸，让织体不精确重复。
- 实际效果：S1 实际增益 = 自动化值（≤0.055×trim≈**0.035**）+ **±0.25** 正弦和——副肺是正身的 ~7 倍（+17dB），慢速搏动，负半周反相仍出声。
- 三条"杀不死"皆由此：trim 只缩放 0.035 项；停止只归零自动化项；**Web Audio 规范：`AudioParam.value` 读数不包含外接信号贡献**——我的无头验证读"全零"时节点仍在出声。

## 3. 施工方自审：为什么三轮都没抓到（方法学失误，如实呈）

1. **我验的是账本，不是声音。** 三轮验证全是读参数自动化值/看调度器状态/查语法——从未渲染过一帧真实音频。最响的声源恰好走在所有仪表之外。
2. **修得快、听得慢的回路里，我把"仪表绿"当成了"听感绿"**，三次向船长宣布"修好了请复听"——把机器该做的实听劳动转嫁给了船长的耳朵。四轮下来船长听麻，进度被拖累，此为本轮最大教训。
3. 每轮修的 bug 都是真的（wow 满幅、绕闸、僵尸自动化、移调），有回归测试在册——但"修了真 bug"≠"修了船长报的 bug"。**验收权一直在船长耳朵上，我却没有给自己造一只机器耳朵。**

## 4. 修法（一行级，候裁未动）

- **方案 A（最小）**：呼吸深度改相对——`bg 深度 = 呼吸系数 × 当前 S1 目标`（如 0.15×0.035≈0.005），并把呼吸 Gain 挂进 stop/接管的归零名单。改动 ~3 行。
- **方案 B（更稳）**：LFO 不再直连增益参数，改为调制一个独立的乘法级（gain×gain 串联），呼吸永远是正身的百分比，数学上不可能反相/压不住。改动 ~6 行。
- 两案均不动 sound-params schema；呼吸系数建议入 sound-params（`bed.breathDepth`≈0.10–0.15，相对值）。

## 5. 验收协议升级案（本报告的重点，请裁）

**新增"机器实听"金门（OfflineAudioContext 离线渲染 RMS，无需人耳即可拦本类 bug）：**
1. **停止即静默**：模拟播放→停止，渲染其后 1s，RMS 必须 < −60dBFS。
2. **总闸有效**：trimDb ±12dB 两次渲染，床段 RMS 差 ≥ 10dB。
3. **床响度守设计**：五带各取 30s 渲染，床 RMS 落在 sound-params 设计值 ±3dB 带内。
4. 以上入 golden 与 `cli ear`，**任何"修好了"的宣称先过机器耳朵，再请船长耳朵**——船长实听从"验证工具"回归"验收权威"，不再当仪表用。
